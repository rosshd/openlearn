## Contract 1 - The Judge output schema

The judge is the single highest-leverage component: difficulty, tier, mastery, and move
selection all consume it. It runs **only when the learner's message is an answer to a pending
check** (classify step). It evolves today's metadata-extractor answer fields into an explicit,
richer verdict. Backward-compatible field names (`last_answer_status`, `answer_score`,
`answer_gap`, `answer_hint`) are kept; new fields are added.

### Fields (the `answer_eval` block of the extractor output)

| Field | Type | Meaning | Notes |
|---|---|---|---|
| `last_answer_status` | `"correct" \| "partial" \| "needs_work"` | Existing verdict | unchanged |
| `answer_score` | float 0.0-1.0 | Quality of the answer | 1.0 fully correct, ~0.5 partial, 0.0 wrong. ZPD target band for a *good* check is 0.5-0.7 |
| `answer_kind` | `"recognition" \| "production"` | Was the question recognition (MC / yes-no / pick) or production (explain / apply / derive / paraphrase)? | **new.** Weights evidence: production correct ≫ recognition correct |
| `is_transfer` | bool | Did the question require applying the concept in a *novel* context vs. reproducing what was shown? | **new.** Required for mastery (Contract 3) |
| `misconception` | string \| null | The specific *wrong mental model* the answer reveals (not just the missing term) | **new.** e.g. "thinks `dd` copies instead of cuts". Distinct from `answer_gap` |
| `answer_gap` | string \| null | Prerequisite concept or misunderstood term | existing. Only when partial/needs_work |
| `gameable` | bool | Could this exact answer have been produced by copying the just-shown text? (judge's *semantic* view) | **new.** Complements the pure-logic detector in Contract 2; either can raise suspicion |
| `answer_hint` | string \| null | One Socratic guiding question (no answer leak) | existing. Only when needs_work |

### Example output

```json
{
  "current_focus": "Vim delete commands",
  "last_answer_status": "partial",
  "answer_score": 0.5,
  "answer_kind": "production",
  "is_transfer": false,
  "misconception": "believes dd deletes a character, not a whole line",
  "answer_gap": "line-wise vs character-wise operators",
  "gameable": false,
  "answer_hint": "What does the doubled key in `dd` signal about the unit it acts on?"
}
```

### Rules / invariants
- When the message is **not** an answer, the judge fields are all omitted (no stale writes -
  this is also the Phase-1 `fresh_score` guard generalized to the whole eval block).
- `answer_score` only valid when `last_answer_status` is set.
- `misconception`/`answer_gap` only when status is partial/needs_work.
- The judge must never contradict a stored MC answer key (existing rule).
- **Calibration is a first-class requirement.** A score of 0.6 must mean the same thing across
  topics; this is tested by the judge-calibration eval (Phase 4), not asserted here.

### Storage
- Persist `last_misconception` and merge `misconception` into a per-concept
  `misconceptions` list on the concept record (Contract 3).
- `answer_kind` / `is_transfer` / `gameable` are transient inputs to the update step; they
  feed the gaming detector and mastery gate, and need not all persist.

## Contract 2 - Gaming-detection rule (pure logic)

Detects answer-extraction without reasoning (copying shown text, hint-cycling, fast-guessing).
**Pure logic, no LLM.** Lives next to `update_learning_metadata`; tunable constants in
`constants.py`.

### Inputs (v1 - two signals)
| Input | Source | Notes |
|---|---|---|
| `overlap` | computed: similarity between learner answer and recently-shown text window | token-trigram Jaccard (0-1) |
| `gameable` | judge (Contract 1) | semantic "could be copied from shown text" flag |
| `answer_kind` | judge (Contract 1) | overlap applies to **production** answers only |
| `answer_tokens` | computed | skip overlap for very short answers |

"Recently-shown text window" = the tutor's prior turn(s) rendered before this answer.

**Deferred to the fine-tuning phase:** `latency_s` (needs REPL input-loop timing plumbing)
and `help_before_attempt`. v1 ships on overlap + the judge flag and degrades gracefully;
adding the timer later only tightens detection. (Per review: keep v1 to the two cheap signals;
add the question timer when we begin tuning the tutor.)

### Starting constants (eval-tuned)
```
GAMING_OVERLAP_TRIGRAM_JACCARD = ~0.6   # answer this similar to shown text is suspect
GAMING_MIN_ANSWER_TOKENS       = 6      # below this, skip overlap (too short to judge)
```

### Decision (v1)
```
gaming_suspected =
    (answer_kind == "production" and answer_tokens >= GAMING_MIN_ANSWER_TOKENS
        and overlap >= GAMING_OVERLAP_TRIGRAM_JACCARD)
    or judge.gameable
```
- Recognition (MC, single letter) is **exempt** from overlap (legitimately short).
- The signal is advisory, never punitive (see response).

### Response (the important part - non-punitive, self-correcting)
If `gaming_suspected` **and** `last_answer_status == "correct"`:
1. Do **not** mark the concept known / do **not** advance.
2. Lower stored confidence in that concept.
3. Set a `pending_verify` state: the next tutor turn poses a **transfer question** on the
   same concept (novel context, ungameable).
   - Learner passes the transfer → suspicion cleared, mastery credited normally.
   - Learner fails → normal `needs_work` / struggling path; no penalty for the false alarm.

This makes a noisy heuristic safe: a genuinely fluent learner simply answers one extra
transfer question and proceeds.

## Contract 3 - Mastery gate (when a concept / unit advances)

Encodes "reproduction ≠ comprehension." Mastery is earned by *transfer*, not by a single
correct answer or self-reported confidence (Bloom mastery; Calibration; Gaming).

### Per-concept record (extends `concept_attempts`)
```json
"concept_attempts": {
  "Vim delete commands": {
    "attempts": 4,
    "correct_sum": 2.5,
    "passed_transfer": true,        // new: ever passed a production+transfer check
    "recognition_only": false,      // new: all correct answers so far were recognition
    "misconceptions": ["dd deletes a char"]  // new: accumulated, cleared on mastery
  }
}
```

All thresholds below are **profile-dependent** - set by the learner's mastery profile
(Contract 4), not hard-coded. The structure is fixed; the numbers and whether transfer is
required move with the goal.

### Concept mastery - `concept_is_mastered(record, profile) -> bool`
1. `answer_score >= profile.mastery_score` on a qualifying correct answer.
2. If `profile.transfer_required`: `passed_transfer == true` (a correct answer with
   `answer_kind=="production"` **and** `is_transfer==true`). If not required, transfer is a
   bonus and a strong production answer suffices.
3. If `profile.recognition_counts == false`: recognition-only correctness does not qualify.
4. Not currently under gaming suspicion for this concept.
5. Rolling accuracy `correct_sum/attempts >= profile.mastery_rate`.

A single fast correct never qualifies under any profile (anti-gaming floor).

### Unit advancement - `unit_is_complete(metadata, unit, profile) -> bool`
`current_unit` increments only when a proportion `profile.unit_mastery_fraction` of the unit's
core concepts are mastered.

**`chapter_complete` is removed from the advancement path entirely** (per review). The mastery
gate is the sole advancement mechanism. If the LLM still emits `chapter_complete`, it is
ignored. This is the single biggest behavioral change from v0.6.0, where it could advance
directly - advancement now reflects demonstrated mastery, not the model's say-so.

### Constants - see Contract 4 (profile table). Defaults (`proficient`): score 0.8, rate
0.75, unit fraction 0.8, transfer required, recognition does not count.

## Contract 4 - Mastery profile (goal-driven rigor)

**Rationale.** Our target demographic spans very different goals for the *same material*. A
biology student who wants a good test grade for the least time spent, and a biology researcher
who needs deep, transferable understanding, should not be held to the same bar. The engine
must let the learner choose the speed↔depth tradeoff; Contracts 2-3 and the move policy are
all parameterized by it. This makes "maximal progress per time spent" a per-learner setting
rather than one hard-coded definition of mastery.

### Profiles
| Profile | For | Mastery bar | Behavior |
|---|---|---|---|
| `efficient` | exam prep, breadth, time-boxed | recall + one application; transfer is bonus; recognition can count | advance fast; fewer impasse probes; don't belabor edge cases |
| `proficient` *(default)* | solid working understanding | passed production + transfer check | balanced; the Contract-3 defaults |
| `deep` | research, foundations, teaching others | production + transfer in *multiple* contexts + explain-back; recognition never counts | more impasse/edge-case probes; interleave across units; explain-back required |

### Threshold table (starting values, eval-tuned)
| Param | `efficient` | `proficient` | `deep` |
|---|---|---|---|
| `mastery_score` | 0.7 | 0.8 | 0.9 |
| `mastery_rate` | 0.7 | 0.75 | 0.85 |
| `unit_mastery_fraction` | 0.7 | 0.8 | 0.9 |
| `transfer_required` | no (bonus) | yes | yes, multi-context |
| `recognition_counts` | yes | no | no |
| impasse-probe frequency (mastering tier) | low | medium | high |

### Anti-gaming floor is profile-independent
Gaming detection (Contract 2) does **not** relax for `efficient`. Copying the shown text is
never learning, regardless of goal - only the *depth bar* moves, never the anti-cheating floor.
An `efficient` learner still must *produce* answers; they just aren't pushed to research depth.

### How it's set & stored (decided: hybrid + course-options override)
- **Hybrid set:** at `openlearn new --goal "..."`, infer the profile by classifying the stated
  goal, then **show the inferred choice and let the learner confirm or override** in one step
  (don't silently pick). Falls back to `proficient` if inference is unclear.
- **Changeable later:** `mastery_profile` is exposed in **course options** so the learner can
  switch profiles mid-course (e.g. cram for an exam now, go `deep` later). Changing it re-reads
  thresholds on the next turn; it does not retroactively un-master concepts.
- Stored as `mastery_profile` in topic metadata; `normalize_topic_metadata` defaults it to
  `proficient` for existing topics.

## Open questions for review
1. ~~Latency capture~~ **Resolved:** deferred to the tuning phase; v1 gaming detection uses
   overlap + judge flag only.
2. **`overlap` metric:** trigram Jaccard is the strawman - is longest-common-word-run more
   robust against paraphrase? (Worth one eval comparison.)
3. **"Core concepts" of a unit:** do we have a clean per-unit concept list, or do we derive it
   from `concept_attempts` keys seen while in that unit?
4. ~~Self-reported confidence~~ **Resolved:** defer explicit confidence prompts until either
   (a) a low-effort UI makes submitting one near-frictionless, or (b) we attach it to completed
   quizzes rather than every turn. Reserve a `confidence` field in the learner model now so
   adding it later isn't a migration. In the meantime, harvest calibration *implicitly* from
   "predict before I show you" moves (mastering tier). Quizzing direction (see
   LEARNING_SCIENCE.md "Test-Enhanced Learning"): low-stakes cumulative spaced quizzes
   interspersed through learning - **not** surprise/graded pop quizzes - likely a Phase 3/4
   mechanic that also serves the unit mastery check and the delayed-retrieval metric.
5. ~~Profile classification~~ **Resolved:** hybrid - infer from the goal, show the choice for
   confirm/override at `new`, and expose it in course options to change mid-course.
6. Are the starting constants/profile thresholds in the right ballpark before the eval loop
   tunes them?

## Phase sequence (re-scoped, per TUTOR_INTERACTION.md §9)
- **Phase 2** - the four contracts: judge fields, v1 gaming detection (overlap + judge flag),
  the mastery gate, and the mastery profile. **Plus the foundations below** (concept identity,
  data boundary, event logging) - they are cheap now and expensive later. Logic + data + tests.
- **Phase 2c - Concept lists (unblocks the mastery layer).** Phase 2b shipped, but unit
  concepts are title-derived (one concept = the unit title), so when the extractor sets a
  finer `current_focus`, practice accrues under a non-unit id and `unit_is_complete` (which
  checks the unit's title-concept) rarely fires - the mastery gate, auto-advance, and profiles
  are largely inert in realistic use. Fix: generate a real per-unit concept list in the outline
  (`course_outline_prompt` emits `concepts` per unit) and attribute an unmatched `current_focus`
  to the *current unit's* concept set. Add tests for the finer-focus case and a leak-guard test
  (dynamic keys never appear in the `.md`). Other carry-forwards to fold in: eval should use the
  production `update_prompt` and fail soft on a missing `answer_score` (C3); confirm whether
  auto-advance should leapfrog remaining slides (C2).
- Carry-forward from 2c review (fold into a later phase): `unit_is_complete` uses the fraction
  of *practiced* concepts, so a multi-concept unit can complete after mastering only the subset
  that was checked, ignoring never-practiced concepts (bounded by the last-slide gate, but not
  rigorous). Refinement: mark outline-defined concepts as "core" (required for completion) vs
  emergent appended ones. Also: guard the judge-calibration eval against zero scored cases.
- **Phase 3** - wire the state→move policy into `select_check_mode` + tutor-prompt fragments
  (ungameable questions, elicit-before-tell, impasse probes, attempt-gated help).
- Phase 3 review carry-forwards (fold into Phase 4 or a quick 3.1 before adding more prompt
  content): **P3-1** consolidate the three overlapping tier/move blocks (`_difficulty_tier_prompt`,
  `check_mode_prompt`, `state_move_policy_prompt`) into one coherent section - they currently
  restate worked-example/mastering guidance ~3× (token cost + dilution). **P3-2** the always-on
  "elicit, do not tell" may under-teach genuinely new material (novices need worked examples
  first); validate via Phase 5 evals and consider scoping it to practice/review, not first
  exposure. **P3-3** move `ROLLING_PASS_RATE_WINDOW` to `constants.py`.
- **Phase 4 (shipped)** - **Cumulative quiz / retrieval mechanic.** Low-stakes, ungraded-by-framing
  quizzes triggered by *accumulated material + spacing + SRS due-density* (not chapter
  boundary); cumulative and interleaved across recently studied units; transfer questions.
  Serves three jobs at once: spaced retrieval practice, the unit-level mastery check
  (Contract 3), and the delayed-retrieval outcome signal. Profile-tuned frequency/format/depth
  (Contract 4). Replaces chapter-complete quizzing. (See LEARNING_SCIENCE.md "Test-Enhanced
  Learning".)
- **Phase 5 (shipped)** - eval loop: move-policy evals + expanded judge-calibration; delayed-retrieval north-star metric.
  *(Judge-calibration eval was pulled forward to accompany the Phase-2 judge - see Foundations.)*
  - **Delayed-retrieval metric** (`delayed_retrieval_metric`, `delayed_retrieval_metric_from_event_log`):
    aggregates spaced-retrieval pass rates from the JSONL event log.
    Counts only retrievals where at least `min_spacing_days` elapsed between first exposure
    and the retrieval attempt.
    Returns `{attempts, passed, pass_rate, by_concept}`.
    Helper functions: `event_retrieval_source` (reads `source`, `retrieval_type`, or `context`
    fields; returns `"review"`, `"srs"`, `"quiz"`, `"cumulative_quiz"`, or `"retrieval"`),
    `event_passed_retrieval` (status `"correct"` or score >= 0.8),
    `spaced_retrieval_items_from_event` (extracts `(concept_id, passed, source)` tuples from
    `answer_judged` and `quiz_completed` events).
  - **Move-policy evals** (`tests/evals/test_move_policy.py`): slow-lane pytest scenarios
    (marker `pytest.mark.slow`; skipped in `OPENLEARN_MOCK=1` mode and when no API key is
    present) that feed curated learner states and messages to the live tutor and score responses
    against rubrics with an LLM judge.
    Pass threshold: 75% of scenarios (`MOVE_EVAL_THRESHOLD = 0.75`).
    Fixtures live in `tests/evals/fixtures/move_eval_scenarios.json`.
  - **Judge-calibration expansion** (`tests/evals/fixtures/judge_calibration_cases.json`):
    labeled cases now include `answer_kind` (`"recognition"` / `"production"`) and `gameable`
    ground-truth labels in addition to the existing `true_score`.
    Test (`tests/evals/test_judge_calibration.py`): MAE <= 0.3 on `answer_score`,
    field accuracy >= 0.7 on `answer_kind` and `gameable`.
  - **P5-1 fix** - `update_learning_metadata` was silently dropping review and SRS re-test
    answers from the delayed-retrieval metric because it never tagged them.
    Fix: `answer_judged` events now include `source` and `is_retrieval: true` when
    `is_review_session` is true (source `"review"`) or the answered concept was SRS-due at
    answer time (source `"srs"`); review-session tagging takes precedence over SRS tagging.
    The `due_review_matches_answer` helper checks concept-level SRS due state.
- **Phase 6** - `/difficulty` visibility & override (demoted).
- **Future** - distill the judge onto a small fine-tuned model (gated on the eval loop + usage
  data).

## Foundations to settle before piling on features

The *pedagogy* research is sufficient to start. The risks that could bottleneck us later are
**architectural / data-shape decisions**, not more studies. Resolve these in/around Phase 2,
because they are cheap now and very expensive to retrofit:

1. **Concept identity (was Open Q3 - promote to a Phase-2 decision).** Per-concept mastery,
   misconceptions, interleaving, and cumulative quizzes all key off "concepts," which are
   currently ad-hoc free-text strings (`concept_attempts` keys, `current_focus`). If the LLM
   emits slightly different strings for the same idea, the per-concept model *fragments* and
   everything downstream silently degrades. Decide a canonical per-unit concept list (tied to
   `course_units`) with stable IDs, vs. continued normalized free-text. **Recommendation:
   canonical concept IDs per unit.** This is the highest-risk foundation.
2. **Data boundary: human-owned content vs. machine learner-model.** Everything lives in one
   editable Markdown file's JSON frontmatter. Rich analytics (per-concept records,
   misconceptions, quiz history, latency, confidence) will bloat it and fight the
   "manually editable, files you own" promise. Decide the boundary now: keep the topic file
   human-facing and move the growing learner-model/analytics to a separate derived store
   (still local, still user-owned). **Recommendation: split them.**
3. **Start an append-only event log now.** The data-flywheel/judge-distillation moat is
   worthless without data captured from day one - *(state, move, judge output, outcome)* tuples
   cannot be backfilled. A simple local JSONL event log is cheap to add in Phase 2 and is what
   makes Phase 5 measurement and the future distillation possible. **Do this in Phase 2 even
   though nothing consumes it yet.**
4. **Pull judge-calibration eval forward.** The judge is load-bearing for everything; don't
   wait until Phase 5 to measure whether it's calibrated. A small labeled (answer → true-score)
   set should land alongside the Phase-2 judge so we don't build four phases on an unmeasured
   component.

Non-bottlenecks (safe to defer): per-role model routing (cheap to add when we distill the
judge), multi-user/schools (local-first already fits), and richer UI.

### Decided foundation design

- **Data split (two files per topic).**
  - **Stable / human-owned:** `learning-topics/<slug>.md` (existing). Holds what rarely changes
    and what a human may edit: `topic`, `slug`, `goal`, `mastery_profile`, course
    scope/excludes/assumptions, `course_units` *structure* (unit, chapter, title, slide_count,
    **canonical `concepts`**), and the body/notes.
  - **Dynamic / machine-owned:** `learning-topics/<slug>.state.json` - the learner-model the
    tutor reads and writes every turn: per-concept records (keyed by concept id), counters,
    rolling pass-rate, per-unit adaptive `difficulty` + `difficulty_locked`, `difficulty_tier`,
    transient answer fields, pending states, quiz history. Still local/user-owned, just not
    meant for hand-editing. New `load_state`/`save_state`; one-time migration lifts the existing
    dynamic fields out of the `.md` frontmatter.
- **Concept identity.** Each `course_units` entry gains `concepts: [{id, label}]` with a stable
  slug `id` assigned at outline-parse time. All per-concept learner data is keyed by `id`. The
  judge attributes an answer to an existing concept `id` (given the unit's concept list in
  context), or proposes a new one to be added.
- **Event log (append-only, forward-compatible).** `learning-topics/<slug>.events.jsonl`. One
  JSON object per line: `{schema_version, ts, event_type, slug, data}`. Never mutated. Emit on
  `answer_judged`, `difficulty_changed`, `gaming_suspected`, `mastery_changed`,
  `unit_advanced`, `quiz_completed`. `schema_version` + opaque `data` mean we can change the
  learner-model shape later without losing or breaking old events - the moat data survives
  refactors.
  Notable `data` fields on `answer_judged` events (relevant to the retrieval metric):
  `source` (`"review"` | `"srs"` | `"quiz"` | absent for regular lesson answers) and
  `is_retrieval: true` are set when the answer was a spaced-retrieval event - either because
  the session is a review session or because the answered concept was SRS-due at answer time.
  These fields are what `delayed_retrieval_metric` and its helpers use to identify retrieval
  items in the log (P5-1).
- **Judge-calibration eval.** A small labeled `(answer → true_score)` fixture set + a slow-lane
  test asserting the judge stays within tolerance. Lands with the Phase-2b judge.

### Phase 2 is split

- **Phase 2a - Foundations (behavior-neutral).** Data split + migration, concept-id model,
  event-log writer, judge-cal eval scaffold. **No pedagogy behavior change; all existing tests
  stay green.** Verify the migration round-trips real topic files.
- **Phase 2b - Contracts (on the clean base).** The four contracts (judge fields, v1 gaming
  detection, mastery gate, mastery profile), reading/writing the dynamic store and emitting
  events.
