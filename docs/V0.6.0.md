# v0.6.0 Plan

Archived historical plan.
Use `AGENTS.md`, `README.md`, and `docs/PLAN.md` for current guidance.

## Goal

Make the tutor noticeably smarter and give learners visible evidence of their progress.
v0.5.0 built the infrastructure (import, SRS, drill sandbox, Rich UI). v0.6.0 makes the
core study loop - question, answer, feedback, adapt - significantly better.

Two headline improvements a user should feel immediately:
1. Feedback explains *why* an answer was wrong, not just that it was wrong.
2. The tutor gets harder as you improve and eases off when you're struggling.

---

## Phase 1 - Structured Evaluation Pipeline

**Goal:** Replace ad-hoc string parsing in answer evaluation with a reliable structured
output contract that the rest of the system can build on.

Current state: `ask_topic` returns a raw string; verdict detection relies on substring
matching ("correct", "needs work") which is brittle and limits what downstream code can do.

- Add a dedicated evaluation call that asks the model to return a small JSON object:
  ```json
  {
    "verdict": "correct | partial | wrong",
    "score": 0.0-1.0,
    "explanation": "one sentence - why this verdict",
    "gap": "prerequisite or concept the learner seems to be missing, or null",
    "hint": "one guiding question to help without giving the answer, or null"
  }
  ```
- Parse this reliably via JSON mode (pass `response_format={"type":"json_object"}` where
  the model supports it) with a regex fallback for providers that ignore it.
- Store `gap` in `weak_spots` and surface it the next time that concept comes up.
- Use `hint` as the first-pass response when `verdict == "wrong"` instead of immediately
  giving the answer - aligns with productive failure research (see LEARNING_SCIENCE.md).
- Keep the tutor's prose response separate from the evaluation JSON; the evaluation is an
  internal signal, not shown verbatim to the learner.
- Tests: structured eval parsing, gap persistence, hint-first wrong-answer flow,
  fallback when model returns prose instead of JSON.

**Stopping point:** Wrong answers always produce an explanation + guiding hint rather than
a generic "needs work." `weak_spots` are populated from structured gaps, not keyword matches.

## Phase 2 - Progress Tracking & Stats

**Goal:** Give learners visible evidence that they're retaining knowledge.

- Session stats stored in topic metadata:
  - `session_count` - number of study sessions
  - `total_study_minutes` - cumulative study time (delta between first and last REPL input per session)
  - `concept_attempts` - `{concept: {attempts: N, correct: N}}` running totals
- Streak tracking in `state.json`:
  - `study_streak` - consecutive calendar days with at least one REPL interaction
  - `last_study_date` - ISO date string; streak resets if > 1 day gap
  - `longest_streak` - all-time record
- Status bar update: show `🔥 N` streak indicator when streak ≥ 2 days (strip emoji for
  non-Unicode terminals via `TERM` check).
- `openlearn stats [topic]` command:
  - If topic provided: Rich dashboard for that topic with streaks, this week's study minutes,
    review forecast, and mastery by unit.
  - If no topic: all-topic Rich dashboard with aggregated event-log activity, review forecast,
    and structured-course mastery.
  - Use `--text` or `--share` for a compact shareable summary.
- Tests: streak increment/reset logic, session timer, accuracy accumulation, stats output
  format.

**Stopping point:** `openlearn stats` renders a one-screen summary with a streak count and
structured mastery details.

## Phase 3 - Adaptive Difficulty

**Goal:** The tutor should feel like it's paying attention to the learner's actual performance.

- Track a `rolling_accuracy` per concept: 5-turn exponential moving average of `score`
  from Phase 1 structured evaluations.
- Difficulty tiers (injected into the tutor system prompt per-concept):
  - **Struggling** (rolling < 0.45): offer worked examples first, use simpler vocabulary,
    break explanations into smaller steps, skip multi-part questions.
  - **On track** (0.45-0.75): current behavior.
  - **Mastering** (rolling > 0.75 for 3+ turns): push free-response over multiple choice,
    ask "why does this work?" and "what breaks if you change X?", suggest adjacent concepts.
- Expose `concept_difficulty_tier` in metadata so future UI can visualize it.
- Don't change tier mid-turn; recalculate at the start of each new question.
- Tests: tier calculation, tier change on score shift, prompt injection per tier,
  no regression to easier questions when streak is just 2 turns.

**Stopping point:** A learner who answers three consecutive questions correctly gets
noticeably harder follow-up questions. A learner who misses repeatedly gets a worked
example instead of another quiz attempt.

## Phase 4 - First-Run Wizard & Course Templates

**Goal:** Get a new user from `pipx install` to first lesson in under 60 seconds.

- `openlearn init` (or auto-trigger on first bare `openlearn` without usable provider config):
  - Check for existing config; skip if already set up.
  - Prompt for provider: OpenAI, Anthropic-compatible, Ollama, or custom OpenAI-compatible.
  - Prompt for API key with masked input via `getpass` when the provider requires one.
  - Prompt for base URL when the selected preset does not declare one; Ollama defaults to `http://localhost:11434/v1`, and OpenAI defaults to `https://api.openai.com/v1`.
  - Prompt for model; OpenAI defaults to `gpt-4.1-mini`, Ollama defaults to `llama3.1`, and custom providers require entry.
  - Validate the connection with `{base_url}/models`; rejected keys can be retried, and network failures can be saved only after confirmation.
  - Write `config.json` on success.
  - Route the learner into Quick Learn, the Vim starter course, or the menu.
- Course templates: curated JSON files shipped in `src/openlearn/templates/`:
  - Each template: `{"name": "...", "goal": "...", "units": [...], "tags": ["cs", "beginner"]}`
  - Ship ~8 templates: Python basics, Git & GitHub, Linux CLI, Algorithms & Data Structures,
    SQL fundamentals, HTTP & APIs, Computer Networks, Vim.
  - `openlearn new <topic> --template <name>` uses the template outline instead of generating one.
  - `openlearn templates` lists available templates with tags.
  - Templates are a starting point - the user still runs outline confirmation flow from v0.2.0.
- Tests: init config write, ping failure handling, template loading, template-seeded outline.

**Stopping point:** `openlearn init` takes < 60 seconds, validates the connection, and writes
a working config. `openlearn new python --template python-basics` starts a structured course
without generating an outline from scratch.

## Phase 5 - DeepEval Conversation Quality Testing

**Goal:** Catch tutor quality regressions automatically, not through manual passhthroughs.

(Deferred from v0.5.0 testing discussion.)

- Add `deepeval` as a `[dev]` dependency.
- Write 4-6 AI-simulated student conversation scenarios:
  - `stuck_learner`: answers wrong 3 times in a row - tutor should offer worked example,
    not just repeat the question.
  - `overconfident_learner`: says "I know this" then answers incorrectly - tutor should
    address the gap without being dismissive.
  - `correct_brief_answer`: learner says "yes" or gives minimal response - tutor should
    probe for understanding before advancing.
  - `correct_full_answer`: detailed correct answer - tutor should advance, not repeat.
  - `off_topic_question`: learner asks something unrelated - tutor should briefly answer
    then redirect to the course.
  - `prerequisite_gap`: answer reveals a missing prerequisite - tutor should identify and
    address it.
- Each scenario uses DeepEval's `GEval` metric with a custom rubric grounded in the
  structured evaluation Phase 1 output.
- Run with `pytest tests/evals/test_conversation_quality.py` (marked slow; excluded from
  default fast test lane to keep CI quick).
- Tests guard against: unhelpful "try again" loops, answer giveaways without hints first,
  topic abandonment when learner is off-track.

**Stopping point:** 6 conversation quality tests pass. CI runs them on a `--slow` flag.
At least one test catches a real regression that the pexpect smoke tests would miss.
