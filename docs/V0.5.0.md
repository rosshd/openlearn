# v0.5.0 Plan

## Goal

Make openlearn feel like a polished, daily-use study tool. v0.4.0 built the core study loop.
v0.5.0 makes it faster, richer, and more capable — better terminal UX, real web import,
PDF support, and the review scheduling that was scaffolded in v0.4.0 brought to full polish.

---

## Phase 1 — Rich UI & platformdirs  ✅

**Goal:** Replace hand-rolled ANSI with `rich`, fix config directory resolution.

- Add `rich` and `platformdirs` to `pyproject.toml` dependencies
- Replace `ui.py` ANSI escape codes, `styled()`, `wrap_prose()`, `format_tutor_line()`,
  `format_tutor_output()` with `rich` equivalents
  - `rich.markdown.Markdown` for lesson/feedback output
  - `rich.syntax.Syntax` for code blocks
  - `rich.console.Console` for all print calls
  - `rich.progress` spinner during LLM streaming calls (replaces silent wait)
  - `rich.panel` / `rich.rule` for section headers and status bars
- Replace `OPENLEARN_HOME` CWD fallback logic with `platformdirs.user_data_dir("openlearn")`
  - Keep `OPENLEARN_HOME` env var as an override for power users
  - Add migration notice if old CWD-based dir is detected
- Update all tests that assert on ANSI output to use `Console(force_terminal=False)` or strip

**Stopping point:** `python -m unittest` passes, `openlearn resume` renders Markdown lesson
output with syntax-highlighted code blocks, spinner shows during streaming, no raw ANSI leaks.

---

## Phase 2 — Review Scheduling Polish  ✅

**Goal:** The v0.4.0 scaffolding (`schedule_review_item`, `due_review_items`) is wired up but
the UX isn't surfaced. Make review a first-class daily habit.

- Show `Reviews due: N` in the status bar when items are due (currently hidden)
- Add `r` as a one-key shortcut in the menu to jump straight to due review
- After a due-review session, prompt `easy / hard / missed` inline — don't wait for a slash command
  - Current v0.5.0 scope treats this as a bulk session grade and reschedules all due concepts
    from that session with the same result.
  - Later polish should support per-concept marking when review sessions cover mixed outcomes.
- `openlearn due` output: use rich table (topic, concept, due date, difficulty)
- Wire `ebisu` as an optional upgrade to the fixed-interval schedule (`easy=7d, hard=2d, missed=1d`)
  — keep fixed-interval as default, ebisu opt-in via config
  - Current implementation is only a scaffold: it uses `defaultModel()` and does not persist
    per-concept Ebisu model state yet. Proper Ebisu support belongs in Phase 5/backlog.
- Add `/review --due` REPL shortcut to start a review session limited to overdue items only
- Tests: review item lifecycle (schedule → due → mark result → reschedule), ebisu path if added

**Stopping point:** A learner can `openlearn resume`, see `Reviews due: 3` in the status bar,
press `r`, answer each item, mark easy/hard/missed inline, and return to the lesson — all without
typing a slug or a long command.

## Phase 3 — PDF & Web Import  ✅

**Goal:** Import real class material — PDFs, web articles, and whole course folders.

- Add `pdfplumber` to deps; extend `cmd_import` to accept `.pdf` files
  - Extract text page by page, join, feed into existing `summarize_context_file` pipeline
  - Print page count and warn if truncated at `CONTEXT_SUMMARY_CHAR_LIMIT`
- Add `requests` + `trafilatura` to deps; add `--url` flag to `cmd_import`
  - `openlearn import <topic> --url https://...` fetches page, extracts clean text, summarizes
  - Reuses existing summarizer — no new prompt logic
  - Warn if page requires JS rendering (trafilatura returns empty; suggest copying text manually)
- Add `--scan <dir>` flag to `cmd_import`
  - Walks directory, imports all `.pdf`, `.md`, `.txt`, `.docx` files in one pass
  - Skips already-imported files by SHA-256 checksum stored in topic metadata
  - Reports: `3 imported, 1 skipped (already imported), 1 failed (bad PDF)`
- Add `python-docx` for `.docx` support in the scan path
- Tests: PDF extraction round-trip, URL import with mocked `requests`, scan deduplication

**Stopping point:** `openlearn import notes --scan ~/class/CS101/` ingests a whole semester
folder in one command, deduplicates on re-run, and the course plan reflects the imported material.

## Phase 4 — Coding Problem Sandbox  ✅

**Goal:** For coding courses, generate and check real exercises without leaving the terminal.

- `/drill` REPL command: tutor generates a `.py` file with:
  - Problem docstring (description, examples, constraints)
  - Function stub
  - `pytest`-style test cases (hidden from the learner until `/check`)
  - Opens file in VS Code via `subprocess.Popen(['code', filepath])`
- `/check` REPL command:
  - Runs `pytest` on the generated file, captures output
  - Feeds pass/fail + error messages back to tutor as a follow-up prompt
  - Tutor gives targeted feedback on the specific failure, not generic hints
- `/drill --leetcode` flag: pulls a real problem from a curated local problem bank
  (small JSON file shipped with openlearn, no external API)
- Problem files saved to `learning-topics/drills/<slug>/` so they're persistent and reviewable
- Tests: file generation format, pytest invocation mock, result parsing

**Stopping point:** A learner in a Python course can type `/drill`, solve a problem in VS Code,
type `/check`, and get specific tutor feedback on what failed — without leaving the study loop.

## Phase 5 — Polish & Hardening  ✅

**Goal:** Fix known gaps found during Phase 1–4 review before adding the final feature layer.

- `openlearn review <topic>` CLI path: pass `input_func=input` so the easy/hard/missed prompt
  fires on direct CLI invocation (currently skipped because `args.func(args)` sends no `input_func`)
  — guard with `sys.stdin.isatty()` so piped/scripted callers are unaffected
- Ebisu per-concept model persistence: store `ebisu_model` alongside each `review_due` item;
  update via `ebisu.updateRecall()` on result; use stored model in `predictRecall()` instead of
  always calling `defaultModel()` — add tests against real ebisu behavior
- Menu happy-path integration test: create → start → answer → `/done` → review (mark easy) →
  switch topic → delete
- `config show` masked key consistency across all output paths
- Topic file `.bak` snapshot before any metadata rewrite in `repair` and `update_learning_metadata`
- `openlearn import <topic> <file>` and `--url` path: call `save_imported_checksum` after success
  so reimporting the same file/URL deduplicates instead of creating duplicate context entries
  (scan already does this correctly)
- `_fetch_url_text`: set a `User-Agent` header on the `requests.get` call to avoid 403s from
  sites that block the default `python-requests/x.x.x` UA
- Interactive menu parity: the new-course setup and Context-files menus now expose file
  (txt/md/pdf/docx), URL, and folder-scan import — previously URL/scan were CLI-only and the
  file option was mislabeled `.txt`. Shared `import_file_source`/`import_url_source` core so the
  CLI and menus follow one import pipeline (dedupe → write → summarize → record checksum).
- `User-Agent` strings across the model API calls and web fetches now track `__version__`
  instead of stale hardcoded `0.1.0`/`0.5.0`.
- `maybe_suggest_videos` only records `last_video_focus` after a successful fetch, so a transient
  network failure no longer permanently silences suggestions for that concept.

**Stopping point:** All known Phase 1–4 gaps closed, 170+ tests passing, no regressions.

## Phase 6 — YouTube Suggestions & Version Bump  ✅

**Goal:** Surface video resources for hard concepts; ship v0.5.0.

- ~~Add `youtube-search-python` to deps~~ — **rejected:** library is unmaintained and breaks
  against modern `httpx`. Implemented a ~40-line `ytInitialData` parser on the existing
  `requests` dep instead (`fetch_video_suggestions` / `parse_video_results`). Zero new deps,
  no API key, AGPL-clean.
- After a `needs_work` or `partial` answer evaluation, `maybe_suggest_videos` appends up to 3 video
  suggestions (title, URL, duration) — gated behind `course_options["suggest_videos"]` (off by
  default, opt-in). De-duplicates via `last_video_focus` so it won't repeat for the same concept.
- `/videos [--n N] [query]` REPL command + `openlearn videos` CLI subcommand: manually trigger a
  search for the current focus (or a supplied query).
- Gracefully degrades if search fails — all paths return `[]`, no crash, nothing shown to learner.
- Bumped version to `0.5.0` in `pyproject.toml` and `__init__.py`; added `openlearn --version`.
- **Remaining:** tag and push `v0.5.0` (manual).

**Stopping point:** 172 tests passing, video suggestions verified against live YouTube, full
menu happy-path integration test covers the core study loop end to end.
