# v0.5.0 Plan

## Goal

Make openlearn feel like a polished, daily-use study tool. v0.4.0 built the core study loop.
v0.5.0 makes it faster, richer, and more capable ‚Äî better terminal UX, real web import,
PDF support, and the review scheduling that was scaffolded in v0.4.0 brought to full polish.

---

## Phase 1 ‚Äî Rich UI & platformdirs

**Goal:** Replace hand-rolled ANSI with `rich`, fix config directory resolution.

- Add `rich` and `platformdirs` to `pyproject.toml` dependencies
- Replace `ui.py` ANSI escape codes, `styled()`, `wrap_prose()`, `format_tutor_line()`,
  `format_tutor_output()` with `rich` equivalents
  - `rich.markdown.Markdown` for lesson/feedback output
  - `rich.syntax.Syntax` for code blocks
  - `rich.console.Console` for all print calls
  - `rich.progress` spinner during LLM streaming calls (replaces silent wait)
  - `rich.panel` / `rich.rule` for section headers and status bars
- Replace `OPENLEARN_HOME` CWD fallback logic with `platformdirs.user_data_dir("openlearn")`
  - Keep `OPENLEARN_HOME` env var as an override for power users
  - Add migration notice if old CWD-based dir is detected
- Update all tests that assert on ANSI output to use `Console(force_terminal=False)` or strip

**Stopping point:** `python -m unittest` passes, `openlearn resume` renders Markdown lesson
output with syntax-highlighted code blocks, spinner shows during streaming, no raw ANSI leaks.

---

## Phase 2 ‚Äî Review Scheduling Polish

**Goal:** The v0.4.0 scaffolding (`schedule_review_item`, `due_review_items`) is wired up but
the UX isn't surfaced. Make review a first-class daily habit.

- Show `Reviews due: N` in the status bar when items are due (currently hidden)
- Add `r` as a one-key shortcut in the menu to jump straight to due review
- After each review item, prompt `easy / hard / missed` inline ‚Äî don't wait for a slash command
- `openlearn due` output: use rich table (topic, concept, due date, difficulty)
- Wire `ebisu` as an optional upgrade to the fixed-interval schedule (`easy=7d, hard=2d, missed=1d`)
  ‚Äî keep fixed-interval as default, ebisu opt-in via config
- Add `/review --due` REPL shortcut to start a review session limited to overdue items only
- Tests: review item lifecycle (schedule ‚Üí due ‚Üí mark result ‚Üí reschedule), ebisu path if added

**Stopping point:** A learner can `openlearn resume`, see `Reviews due: 3` in the status bar,
press `r`, answer each item, mark easy/hard/missed inline, and return to the lesson ‚Äî all without
typing a slug or a long command.

## Phase 3 ‚Äî PDF & Web Import

**Goal:** Import real class material ‚Äî PDFs, web articles, and whole course folders.

- Add `pdfplumber` to deps; extend `cmd_import` to accept `.pdf` files
  - Extract text page by page, join, feed into existing `summarize_context_file` pipeline
  - Print page count and warn if truncated at `CONTEXT_SUMMARY_CHAR_LIMIT`
- Add `requests` + `trafilatura` to deps; add `--url` flag to `cmd_import`
  - `openlearn import <topic> --url https://...` fetches page, extracts clean text, summarizes
  - Reuses existing summarizer ‚Äî no new prompt logic
  - Warn if page requires JS rendering (trafilatura returns empty; suggest copying text manually)
- Add `--scan <dir>` flag to `cmd_import`
  - Walks directory, imports all `.pdf`, `.md`, `.txt`, `.docx` files in one pass
  - Skips already-imported files by SHA-256 checksum stored in topic metadata
  - Reports: `3 imported, 1 skipped (already imported), 1 failed (bad PDF)`
- Add `python-docx` for `.docx` support in the scan path
- Tests: PDF extraction round-trip, URL import with mocked `requests`, scan deduplication

**Stopping point:** `openlearn import notes --scan ~/class/CS101/` ingests a whole semester
folder in one command, deduplicates on re-run, and the course plan reflects the imported material.

## Phase 4 ‚Äî Coding Problem Sandbox

**Goal:** For coding courses, generate and check real exercises without leaving the terminal.

- `/drill` REPL command: tutor generates a `.py` file with:
  - Problem docstring (description, examples, constraints)
  - Function stub
  - `pytest`-style test cases (hidden from the learner until `/check`)
  - Opens file in VS Code via `subprocess.Popen(['code', filepath])`
- `/check` REPL command:
  - Runs `pytest` on the generated file, captures output
  - Feeds pass/fail + error messages back to tutor as a follow-up prompt
  - Tutor gives targeted feedback on the specific failure, not generic hints
- `/drill --leetcode` flag: pulls a real problem from a curated local problem bank
  (small JSON file shipped with openlearn, no external API)
- Problem files saved to `learning-topics/drills/<slug>/` so they're persistent and reviewable
- Tests: file generation format, pytest invocation mock, result parsing

**Stopping point:** A learner in a Python course can type `/drill`, solve a problem in VS Code,
type `/check`, and get specific tutor feedback on what failed ‚Äî without leaving the study loop.

## Phase 5 ‚Äî YouTube Suggestions & Test Hardening

**Goal:** Surface video resources for hard concepts; harden the full v0.5.0 test suite.

- Add `youtube-search-python` to deps
- After a `needs_work` or `partial` answer evaluation, tutor optionally appends 2-3 video
  suggestions (title, URL, duration) ‚Äî gated behind `course_options["suggest_videos"]` flag
  (off by default, opt-in)
- `/videos` REPL command: manually trigger a video search for the current concept
- Gracefully degrade if search fails (no crash, no error shown to learner)
- Test hardening:
  - Menu happy path integration test: create ‚Üí start ‚Üí answer ‚Üí `/done` ‚Üí review ‚Üí switch ‚Üí delete
  - `config show` masked key consistency across all output paths
  - Topic file backup before metadata rewrites (`repair`, `update_learning_metadata`)
- Bump version to `0.5.0` in `pyproject.toml` and `__init__.py`
- Tag and push `v0.5.0`

**Stopping point:** All tests pass, video suggestions work in a real session, and the full
menu happy-path integration test covers the core study loop end to end.
